### 2026-07-22
- **Added** web-editable `serverDZ.cfg`. It was the one config surface the editor could not reach: repo-owned CODE, rendered at deploy time, so changing `maxPlayers` or the day/night cycle meant a repo edit plus a redeploy. It is now a prestart ARTIFACT - `Apply-ServerCfg.ps1` renders it from `serverDZ.cfg.template` (code) + `host.env` (the two passwords, host-local) + `server-settings.json` (box-owned, `web:'patch'`). The template ships UNRENDERED, so the passwords now resolve only on the box and no deploy-time temp file ever holds them. **Closed allowlist of 14 keys** (owner's list): hostname, description, maxPlayers, enableWhitelist, forceSameBuild, disableVoN, vonCodecQuality, disable3rdPerson, disableCrosshair, disablePersonalLight, lightingConfig, serverTimeAcceleration, serverNightTimeAcceleration, serverTimePersistent. Everything else stays template-owned on purpose - `instanceId`/`shardId` select the persistence folder (a change reads as a wipe), `verifySignatures` is a security control, and `class Missions` is load-bearing but map selection is map.env's job. Per-field fail-soft (bad type/range = warn + template default stands), per-file atomic: a missing `host.env`, a leftover `{{placeholder}}`, or a lost `class Missions` block aborts the write and leaves the previous file. Deploy renders it once seed-if-missing so a box without pwsh still boots.
- **Fixed** the day/night cycle formula in `docs/SETUP.md` and `.internal/OPERATIONS.md`. The old one (`full = 24/X`, `night = 24/(X*Y)`, `day = full - night`) charged all 24 in-game hours at the day rate and then subtracted night out of the result, overstating the cycle - and at Y=1 it reported zero daylight instead of an even split. Correct: `day = D/X`, `night = (24-D)/(X*Y)`, `cycle = day + night`, with D = in-game daylight hours (12 assumed). Y is exactly the day:night ratio. The live server (X=5, Y=4) runs a **3h** cycle - 2h24m daylight, 36m night - not the 4h48m the old formula implied.
- **Added** cycle sliders to the ConfigViewer for the `server-settings.json` row: both multipliers as sliders + number boxes, with a live day/night proportion bar and real-time lengths. Writes the same field override as the list below it, so nothing new is needed on the save path. That row is **fields-only** (no View/Edit file tabs - the whole-file editor could add keys the renderer's allowlist refuses), and the two cycle keys are **hidden from the field list** because the panel already owns them; a Reset-to-default button in the panel replaces the row's ✕.
- **Added** a comment column to the field editor, driven by an optional `"_help": { key: text }` map in the file itself. For `server-settings.json` the text is lifted verbatim from `serverDZ.cfg.template`'s trailing `//` comments - the game's own wording, not a second set to maintain - and the gate asserts the two stay identical, so editing a cfg comment can never leave the UI describing old behaviour. Files without `_help` keep the original four-column layout. Underscore keys (`_readme`/`_help`) are no longer offered as overridable fields: the override engine drops them, so an override there silently did nothing.
- **Added** display-only field renaming via an optional `"_labels": { key: "shown name" }` map, used to surface `enableWhitelist` as **`enableAllowList`**. The game dictates the cfg key, so the real one is still what gets written, patched and filtered - `_labels` only changes what a human reads. The key cell carries `data-key` with the true name, the filter matches either (searching "whitelist" still finds it), and the tooltip spells out what actually lands in the cfg. Same intent as the registry already surfacing `whitelist.txt` as "Allowlist". The gate rejects a label on a key that is not a real toggle, so a typo fails loudly instead of silently doing nothing.
- **Changed** override snapshot retention to 30 on BOTH sides. The box snapshots `config-overrides.json` on every web-editor write into `.overrides-versions/` (`snapshot_overrides` in `Api/deploy/templates/dayz-ctl.template`) - that was still 10 while the repo-side `Sync-ConfigOverrides.ps1 -KeepVersions` had moved to 30, so "how far back can I roll an override?" had two different answers. The box count is now a named `OVR_KEEP=30` next to the function instead of a bare `tail -n +11`. Takes effect on the next **Api** deploy, not the DayZ one - dayz-ctl is rendered from that template. `snapshot_spawns` (map-points) and the owned-file versions are deliberately left at 10.
- **Added** an OPEN-server option: leave `DEPLOY_SERVER_PASSWORD` empty in host.env (keep the key, drop the value) and serverDZ.cfg renders `password = "";`. Setting a password is unchanged and fully supported - empty is just the opt-in. The distinction is key PRESENCE: an empty value is deliberate (open), a MISSING key reads as an unconfigured host and refuses (that refusal, with no prior serverDZ.cfg, is what loops the boot). Parsing widened `(.+?)`->`(.*?)` in both Apply-ServerCfg and the deploy; the deploy renders the placeholder through `[string]$DeployServerPassword` so an empty/null value can't throw. `DEPLOY_ADMIN_PASSWORD` must still be non-empty (admin boundary, with SuperAdmins.txt).
- **Note:** map selection is untouched. Nothing has ever written `serverDZ.cfg` from `map.env` - the active map is `map.env` → the unit's `-mission=`. `serverDZ.cfg`'s `class Missions` block must merely EXIST for the engine to run a mission (its `template=` value is overridden by `-mission=`, which is why the live cfg says chernarusplus while the server runs sakhal). The renderer copies that block through verbatim and refuses to write without it; the gate asserts it.
