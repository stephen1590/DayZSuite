### 2026-07-23
- **Added** per-layer map calibration to the ConfigViewer Map tab. ROOT CAUSE it fixes: the renderer placed EVERY tile layer onto the same world rectangle `[0,worldSize]` with one shared transform, so a layer aligned only if its source image already spanned that rectangle - true for the heightmap-derived `terrain` layer (aligned by construction), false for the higher-res rips (`Sat (New)`, `Topo (New)`, `satellite`) whose pixel bounding box differs. There was no per-layer correction, and the old "2-point sync" was a one-time manual nudge, never persisted. Now each layer carries an optional AFFINE (`map-calibration.json`, a COMMITTED sidecar - the manifest is gitignored + regenerated by MapDataExtraction, so calibration can't live there) that maps assumed-world → true-world; the renderer honors it (`drawTileAffine`, inverse-mapped tile culling) and markers stay in true world coords. **Absent/identity calib = the literal pre-existing draw path**, so aligned layers (Topo Old + everything working) cannot regress - proven offline: the calibrated path at identity produces byte-identical tile placement. Author via a **Calibrate mode** (`js/map-calibrate.js` = pure, unit-proven solver; UI in map.js): mark control points (click the landmark on the raw layer, type its true X/Z read off the accurate layer), 2 pts → offset+scale, 3+ → full affine via least-squares, live residual/RMS/rotation readout, Preview toggle, Copy emits the sidecar block. Answer to "how many points": 2 minimum (offset+scale), 3 for rotation, 3-5 spread for robustness - the residual readout tells you if the simpler model sufficed. The solver, the affine inverse, the identity-preservation, and a full cropped+scaled+rotated round-trip are all proven offline (no browser needed for the math); the DOM/canvas interaction is verified by USING it (non-destructive - preview only until you Copy + deploy). Ships with the ConfigViewer deploy (static assets, no Api/DayZ involvement).

### 2026-07-22
- **Added** the StarDZ DayZ Modding Wiki as a git submodule (`StaticishSite/external/DayZ-Modding-Wiki`, pinned `1148527`), mounted into the static site one node BELOW our own docs at `/docs/reference/dayz-modding-wiki/`. Chosen over embedding it in ConfigViewer's Mod Docs (which would have taken that tab from 6 mod readmes to 111 pages) and over a top-level site section (which would bury 6 of our own pages under 105 of someone else's). **Only `en/` is mounted** - the repo carries 12 languages and 44 MB. The submodule is a BUILD-time dependency only: `Deploy-Site` ships `./public`, so the box never sees the source. Second submodule in this repo after `themes/hugo-book`, so `git clone --recursive` was already required.
- **Added** `layouts/_default/_markup/render-link.html`. The wiki cross-references itself with ~1,200 relative `.md` links and Hugo publishes each page as its own directory, so they need both the `.md` stripped AND one extra `../` (the page sits one level deeper than its source file). Two bugs found by actually sweeping the rendered HTML rather than eyeballing it: missing the extra `../` left **948 of 13,804** links dead, and lower-casing paths left another **149** dead because this site sets `disablePathToLower` (Hugo publishes `README/`, not `readme/`). The hook also sends the upstream's 11 sibling-language links and `CONTRIBUTING.md` to GitHub, since only `en/` is mirrored. Final sweep: **0 broken links**.
- **Note** on licence: the wiki is CC BY-SA 4.0 and we now REPUBLISH it on the public apex domain. `content/docs/reference/dayz-modding-wiki/_index.md` carries the attribution, licence link, pinned commit and an accuracy caveat (its serverDZ.cfg chapter documents 25 parameters and omits several real ones). Keep that page if the mirror stays - it is the attribution requirement, not decoration.
- **Added** an "Other links" group at the bottom of the ConfigViewer's Mod Docs tab, driven by `ConfigViewer/web/mod-links.json`. Everything else in that tab is a doc file read off the box out of an `@mod` folder; these are the off-box SOURCES those mods come from, so "where is this documented?" is answered next to the docs instead of in someone's bookmarks. Seven links: Expansion Scripts + Missions, VPP repo + wiki, the StarDZ community wiki, and dzconfig's serverDZ.cfg page + wiki index. Static asset fetched same-origin with no credentials - **ConfigViewer deploy only, no Api or DayZ redeploy**. Adding a link = one row in that JSON. The renderer drops anything whose URL is not an absolute http/https one (parsed with NO base: resolving against `location.href` would turn a typo like `github.com/foo`, or any junk string, into a same-origin link pointing at our own site), and the links render even when the box has no mod docs at all - that is exactly when they matter most.
